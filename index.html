🧩 How JavaScript Works · under the hood

A complete walkthrough — from engine to closures, in my own words

1. The three pillars

Before we dive into engines and stacks, there are three truths that shape everything else. I keep coming back to them whenever I debug weird behaviour.

📌 Single‑threaded — JS has one call stack and one thread. It does one thing at a time. No parallel execution inside the language itself.

📌 Synchronous by default — code runs top to bottom, line after line. A new line doesn't start until the previous one finishes.

📌 Interpreted + JIT‑compiled — modern engines don't just interpret; they use JIT (Just‑In‑Time) compilation. They start fast with bytecode, then compile hot functions into optimised machine code on the fly.

❌ Myth to kill: “JS is multi‑threaded because async works.”
No — async (timers, network, events) is handled by the environment (browser or Node), not the JS engine. We'll see this clearly when we talk about the event loop later.

2. JavaScript engines · the pipeline

JS code can't run in thin air — it needs an engine. Engines are mostly written in C++ and they translate our code into something the machine can execute.

Under the hood, every engine follows a similar pipeline:

Step What happens Output
1. Tokenizing / Lexing Breaks source code into smallest meaningful pieces called tokens. let, x, =, 10, ;
2. Parsing → AST Tokens are organised into a tree structure (Abstract Syntax Tree) that represents the grammar of your code. Nested tree of nodes
3. Compilation (JIT) Ignition (interpreter) turns AST into bytecode and executes it. TurboFan (optimiser) monitors hot code and recompiles it into fast machine code. Bytecode → optimised machine code
4. Execution The generated bytecode / machine code runs on the call stack. Program output

3. Execution Context · the container

Every piece of JS runs inside an Execution Context (EC). It's like a sealed box that holds variables, functions, and the value of this.

Global EC (GEC)

Created automatically when the program starts. There's only one GEC. In browsers, it creates the window object and sets this to window. In Node.js, it creates global.

Function EC (FEC)

A new FEC is created every time a function is called. Each call gets its own private space. When the function returns, its EC is usually destroyed (closures are the special case).

⚡ Two phases of every EC — this is crucial:

1. Creation phase (Memory allocation) — JS scans the code and sets up memory: var variables are set to undefined, function declarations are stored fully, and this is determined.

2. Execution phase (Code runs) — code runs line by line, assigning real values to the variables that were reserved in phase 1.

Breakdown of the two phases for a simple script
Phase What happens to var x What happens to function greet() this binding
Creation (Memory) Memory allocated, set to undefined Entire function stored in memory Set to window (browser) / global (Node)
Execution (Code) Assigned actual value (e.g., 10) Function is ready to be invoked Used when function is called
// creation phase: x → undefined, greet → function stored console.log(x); // undefined var x = 10; greet(); // "Hello" works (function hoisted) function greet() { console.log("Hello"); }

4. Call Stack · LIFO in action

The call stack is a LIFO (Last In, First Out) structure that tracks which function is currently running. When a function is called, it's pushed; when it returns, it's popped.

function one() { two(); console.log("one done"); } function two() { three(); console.log("two done"); } function three() { console.log("three done"); } one();
Step Action Stack (top → bottom)
1one() calledone → GEC
2two() called inside onetwo → one → GEC
3three() called inside twothree → two → one → GEC
4three finishes → printstwo → one → GEC
5two finishes → printsone → GEC
6one finishes → printsGEC

Output order:

three done two done one done

Stack overflow — if functions keep getting pushed without popping (infinite recursion), the stack blows up and you get Maximum call stack size exceeded.

5. Hoisting · the creation‑phase illusion

Hoisting is the behaviour where declarations appear to move to the top. Nothing physically moves — it's just the creation phase allocating memory before execution.

Declaration type Hoisted? Initial value Access before declaration
var ✅ Yes undefined Returns undefined (no error)
let / const ✅ Yes (but in TDZ) Not initialised ❌ ReferenceError (Temporal Dead Zone)
Function declaration ✅ Yes (full body) Entire function ✅ Works perfectly
Function expression ⚠️ Only variable hoisted Variable set to undefined ❌ TypeError (not a function)

var — hoisted as undefined

console.log(a); // undefined (no error) var a = 5; console.log(a); // 5

let & const — Temporal Dead Zone (TDZ)

They are hoisted, but they are not initialised. From the start of the block until the declaration line, they are in the TDZ — accessing them throws a ReferenceError.

console.log(b); // ❌ ReferenceError: Cannot access 'b' before initialization let b = 10;

Function declarations vs expressions

Function declaration — fully hoisted (body included).

sayHi(); // ✅ "Hi" function sayHi() { console.log("Hi"); }

Function expression — only the variable is hoisted, not the assignment.

sayBye(); // ❌ TypeError: sayBye is not a function var sayBye = function() { console.log("Bye"); };

6. Scope · where variables live

Scope defines the region where a variable is accessible.

Scope type Where declared Accessible from var behaviour let / const behaviour
Global Outside any function/block Everywhere Global Global
Function Inside a function Only inside that function Function‑scoped Function‑scoped
Block Inside { } (if, for, while) Only inside that block ❌ Ignores block (leaks out) ✅ Block‑scoped
{ let a = 1; var b = 2; } console.log(b); // 2 (var leaked) console.log(a); // ❌ ReferenceError

Lexical scope

Lexical means “based on where the code is written” — not where it's called. A function's scope is determined by its physical location in the source code.

7. Scope Chain · the outward journey

When JS sees a variable, it searches in this order:

  1. Current scope
  2. Parent (outer) scope
  3. Parent's parent … all the way up to global
  4. If still not found → ReferenceError

This is the scope chain. It only goes outward/upward, never inward.

Variable Scope where found Lookup path
c Current (inner function) Found immediately
b Parent (outer function) inner → outer → found
a Global inner → outer → global → found
z (not declared) Not found anywhere inner → outer → global → ❌ ReferenceError
let a = 10; function outer() { let b = 20; function inner() { let c = 30; console.log(a, b, c); // 10 20 30 } inner(); } outer();

8. Closures · the superpower

A closure is a function that remembers its lexical environment even after the outer function has finished executing. Think of it as a “backpack” — the function carries the variables from where it was born.

function counter() { let count = 0; return function() { count++; return count; }; } const inc = counter(); console.log(inc()); // 1 console.log(inc()); // 2 console.log(inc()); // 3

count stays alive because the returned function holds a reference to it. That's the closure.

Closure use cases with examples
Use case Description Example
Data privacy Create private variables that can't be accessed from outside bankAccount.balance → private
Function factories Generate customised functions with preset parameters multiplier(2) → double function
Counters Each counter gets its own independent state c1() and c2() have separate counts
Currying Transform multi‑argument functions into chains of single‑argument functions add(1)(2)(3) → 6

The classic loop + closure trap

// using var → shared i for (var i = 0; i < 3; i++) { setTimeout(function() { console.log(i); }, 1000); } // output: 3 3 3 (not 0 1 2)

Why? var is function‑scoped, so there's only one i shared across all iterations. By the time the timeouts run, i is already 3.

Approach How it works Result
Using let Block‑scoping creates a fresh i for each iteration ✅ 0 1 2
IIFE (old‑school) Immediately invoked function captures current value as parameter ✅ 0 1 2

Fix with let — block‑scoping gives each iteration its own i.

for (let i = 0; i < 3; i++) { setTimeout(() => console.log(i), 1000); } // 0 1 2 ✅

Old‑school fix (IIFE) — captures the current value.

for (var i = 0; i < 3; i++) { (function(j) { setTimeout(() => console.log(j), 1000); })(i); } // 0 1 2 ✅

🧠 Key insight: closures capture variables, not values. And scope (var vs let) decides how many variables actually exist. Understanding this changes how you write loops, callbacks, and async code.
